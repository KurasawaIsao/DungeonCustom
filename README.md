# 不思議のダンジョンCustom

C++ / DirectX 11 で制作している、ローグライク形式の 3D ダンジョンゲームです。
ターン制の移動・戦闘、ランダムダンジョン生成、アイテム、罠、敵 AI、ショップやモンスターハウスなどの特殊部屋を実装しています。

# 実行動画
https://youtu.be/4QNcqaHmSag

## 主な実装内容

- 部屋と通路を組み合わせるダンジョン自動生成
- プレイヤー、敵、仲間ユニットを扱うターン制システム
- 追跡、巡回、逃走、距離維持などの敵 AI
- アイテム、杖、食料、状態異常、罠のデータ駆動管理
- ショップ、モンスターハウス、階段、祠などのマップオブジェクト
- ImGui を使ったダンジョン、マップ、出現テーブル編集機能
- DirectX 11 による 3D モデル、UI、ミニマップ、エフェクト描画

## 使用技術

- C++
- DirectX 11
- Visual Studio 2022
- ImGui
- nlohmann/json
- Assimp

## ビルド方法

1. Visual Studio 2022 で `Project1.sln` を開きます。
2. 構成を `Debug` または `Release`、プラットフォームを `x64` に設定します。
3. ビルドして実行します。

## 補足
`.vs/`、`x64/`、中間オブジェクト、ユーザー個別設定などの生成ファイルは Git 管理から除外しています。
ソースコード、ゲームデータ、シェーダー、モデル・テクスチャなど、プロジェクトの確認に必要なファイルを保存対象にしています。


## 内部コード概要
シングルトンについて<br>
MapManagerやUnitManagerのポインタ参照や、MessageLogやLightManagerといった実体参照がありますが、
すべてのシーンで使う予定があり、常駐する必要があるかないかで設計方法としてポインタと実体を分けています。

ゲーム進行関係:<br>

-敵AI切り替えについて<br>
 主にEnemyのApplyData()にて行っていますが、<br>
 データベースにおいて設定したAIの内容に基づき、Enemyが所持しているUnitAIに<br>
 対応したAI(UnitAI)を入れます。<br>
 以降、基本は入れたAI->Update(*this, MapData)で振る舞いを変えて処理します。<br>

-マップ管理について<br>
 MapDataにてマップの状態に関する操作が行えますが、<br>
 MapData自体はMapManagerの保護下にあるため、<br>
 MapManager::GetCurrentMap()->と経由して操作することになります。<br>
 また、ダンジョンのデータや現在の階層データもMapManagerが保持するためここでもMapManagerを経由します。<br>

-ターン管理について<br>
 ターン進行はTurnManagerが担当します。<br>
 基本的な流れは、PlayerTurnでプレイヤーの入力と行動を処理し、<br>
 行動が完了したらEnemyAction、EnemyMove、MoveResolutionを経由して<br>
 再びPlayerTurnへ戻る形になります。<br>
 敵や仲間の行動は、攻撃などの行動フェーズと移動フェーズを分けて処理しています。<br>
 これにより、倍速や三倍速のユニットが持つ行動回数・移動回数を個別に扱えるようにしています。
 また、移動アニメーション中や投擲物の処理中は次のフェーズへ進めず、<br>
 演出が完了してからターンを進めます。<br>
 そのため、ゲーム内部のマス移動と画面上の見た目がずれないようになっています。<br>

-ユニット管理について<br>
 プレイヤー、敵、仲間の参照管理はUnitManagerが行います。<br>
 UnitManagerは現在存在している敵や仲間の一覧を保持し、<br>
 指定座標にユニットがいるか、隣接する敵がいるか、<br>
 プレイヤーや敵への参照取得などを担当します。<br>
 移動や攻撃時にはUnitManager::GetUnitAt()を使って、<br>
 移動先に他ユニットがいないか、攻撃対象が存在するかを確認します。<br>
 これにより、各Unitが個別に全ユニットを探すのではなく、<br>
 UnitManagerを経由して盤面上のユニット状態を確認する形にしています。<br>

-ダンジョン開始と階層遷移について<br>
 DungeonSceneの初期化時にMapManagerへDungeonDataを渡し、<br>
 MapManager::StartDungeon()で現在階層のFloorDataをもとにマップ生成を行います。<br>
 生成後は敵、アイテム、プレイヤーの配置を行い、<br>
 MapRendererにBuildを呼ぶことで画面表示へ反映します。<br>
 階段や効果によって次の階層へ進む場合は、MapManager::RequestNextFloor()を呼びます。<br>
 この中で現在階層番号を進め、演出としてFloorTransitionUIを挟んでから<br>
 MapManager::ChangeFloor()で新しい階層を生成します。<br>
 階層移動時には、現在の敵やマップ上のオブジェクトを破棄し、<br>
 次のFloorDataに基づいて新しいMapDataを作成します。<br>
 ダンジョンの最終階層を超えた場合はResultシーンへ遷移します。<br>

-マップ生成と配置について<br>
 MapManagerが現在階層のFloorDataを参照し、<br>
 ランダム生成か固定マップ読み込みかを判断します。<br>
 ランダム生成の場合はMapGeneratorを使用し、<br>
 固定マップの場合はMapLoaderでJSONからMapDataを復元します。<br>
 敵、アイテム、罠などの配置数や出現内容はFloorDataに設定された値や<br>
 各種出現テーブルIDをもとに決定します。<br>
 そのため、階層ごとの難易度や出現傾向はFloorData側で調整でき、<br>
 実際の生成処理はMapGenerator側に集約しています。<br>

-移動後処理について<br>
 ユニットが移動した後の処理はMapManager::AfterUnitMoved()で行います。<br>
 主にプレイヤーが移動した後、現在いる部屋、店内状態、足元のオブジェクト、<br>
 モンスターハウスの発見などを確認します。<br>
 足元にアイテム、罠、祠などのMapObjectがある場合は、<br>
 そのオブジェクトのOnStepped()を呼びます。<br>
 これにより、プレイヤー側ではなく各オブジェクト側に<br>
 踏まれた時の処理を持たせる形にしています。<br>

-アイテム・罠などの効果処理について<br>
 アイテム使用、罠の発動、スキルなどの効果はEffectBaseを継承したクラスで処理します。<br>
 効果を発動する際はEffectContextに使用者、対象、座標、方向、<br>
 効果の種類やランクなどを入れてApply()を呼びます。<br>
 これにより、回復、ダメージ、ワープ、吹き飛ばし、AI変更などの処理を<br>
 共通の形で扱えるようにしています。<br>
 アイテムや罠は効果の発動元が違うだけで、<br>
 最終的な処理はEffectBase系のクラスに任せる設計になっています。<br>

-自然湧きについて<br>
 ターンが一定数進むとTurnManager::SpawnEnemy()によって敵の自然湧きが行われます。<br>
 通常時は一定ターンごとに敵を追加し、<br>
 店での泥棒状態など特殊な状態では湧き間隔や上限を変えています。<br>
 自然湧きの敵も現在階層のFloorDataを参照して決定されるため、<br>
 階層ごとの敵出現テーブルに沿った敵が追加されます。<br>

エディタ関係:<br>

-エディタシーンについて<br>
 EditorSceneでは、編集用のCamera、MapRenderer、MapEditorを生成します。<br>
 実際の編集処理はMapEditorが中心となって行い、EditorScene自体は<br>
 編集用のシーンを用意する役割に留めています。<br>

-MapEditorについて<br>
 MapEditorは、編集中のMapDataとDungeonDataを保持し、<br>
 ImGuiを使って地形編集、オブジェクト配置、ダンジョン階層設定、<br>
 出現テーブル編集を行います。<br>
 ゲーム本編ではMapDataはMapManagerの管理下にありますが、<br>
 エディタでは編集中の一時データとしてMapEditorがMapDataを直接保持しています。<br>
 これは、エディタ上ではまだゲーム進行中の正式なマップではなく、<br>
 保存前の編集データとして扱うためです。<br>

-地形編集について<br>
 地形編集はMapGeometryEditorが担当します。<br>
 床、壁のタイル変更や、部屋情報の更新を行い、<br>
 変更後はMapRendererにBuildを呼ぶことで見た目を同期します。<br>
 MapRendererはあくまで表示用であり、<br>
 マップの状態そのものはMapData側に保持されます。<br>
 そのため、編集内容を保存する際もMapRendererではなくMapDataを参照します。<br>

-オブジェクト配置について<br>
 アイテム、罠、祠などの配置はMapObjectPlacerが担当します。<br>
 配置されたオブジェクトはEditorPlacedObjectとして保持し、<br>
 種類、ID、座標、個数や杖の回数などの追加情報を持たせています。<br>
 固定マップとして保存する場合は、<br>
 タイル情報だけでなく配置オブジェクトもまとめてJSONに出力します。<br>
 読み込み時にはJSONからMapDataを復元し、配置オブジェクトも再生成します。<br>

-マップファイル管理について<br>
 マップデータの保存と読み込みはMapFileManagementを経由します。<br>
 MapEditor側ではファイル名や保存先の選択を行い、<br>
 実際のJSON出力・入力処理はMapFileManagementに任せています。<br>
 保存先は用途ごとに分けており、<br>
 通常の固定マップはDungeonData\\EdittedMapData、<br>
 部屋パーツはDungeonData\\EdittedRoomDataに保存します。<br>
 これにより、1フロア全体として使うマップと、ランダム生成中に差し込む部屋パーツを区別しています。<br>

-ダンジョン階層編集について<br>
 ダンジョン全体の構造はDungeonDataとして管理します。<br>
 DungeonStructureEditorでは、階層ごとのFloorDataを編集し、<br>
 マップサイズ、部屋数、敵・アイテム・罠の出現テーブル、<br>
 視界距離、特殊部屋の出現率などを設定します。<br>
 各階層のマップ生成方法は、ランダム生成と固定マップのどちらかを選べます。<br>
 ランダム生成の場合はFloorDataの設定をもとにMapGeneratorがマップを作成し、<br>
 固定マップの場合は指定されたJSONファイルをMapLoaderが読み込みます。<br>
 また、複数階層に同じ設定を反映しやすくするため、<br>
 選択した階層の内容を他の階層へ一括適用する機能も用意しています。<br>
 これにより、似た構造の階層をまとめて作り、必要な部分だけ後から調整できます。<br>

-出現テーブル編集について<br>
 敵、アイテム、罠の出現内容はSpawnTableEditorで編集します。<br>
 各テーブルはIDで管理し、FloorDataには使用するテーブルIDだけを保存します。<br>
 実際の生成時には、MapManagerが現在階層のFloorDataを参照し、<br>
 EnemyTableDatabase、ItemTableDatabase、TrapTableDatabaseから<br>
 対応する出現テーブルを取得します。<br>
 これにより、階層データ側は「どのテーブルを使うか」だけを持ち、<br>
 具体的な出現内容は各データベース側で管理できます。<br>

-テストプレイについて<br>
 エディタからは作成したダンジョンデータを指定してテストプレイできます。<br>
 テストプレイ開始時にはDataReferenceに次に読み込むダンジョンIDや<br>
 エディタテスト中であることを記録し、DungeonSceneへ遷移します。<br>
 DungeonScene側ではDataReferenceの情報を参照してDungeonDataを読み込み、<br>
 MapManagerに渡して通常のゲーム進行と同じ流れでダンジョンを開始します。<br>
 そのため、エディタで作成したデータを本番に近い状態で確認できます。<br>
 また、テストプレイ時にはランダムシード(Use Generation Seed)を固定するか、<br>
 毎回変化させるかを選べるようにしています。<br>
 固定すれば同じ構造を再確認しやすくなり、<br>
 ランダム化すれば生成パターンの確認に使えます。<br>

-生成AIの利用について<br>
 本プロジェクトでは、設計方針の整理、コード構成の見直し、コメント文や内部資料の作成補助、<br>
 不具合原因の調査補助などに生成AIを利用しています。<br>
 ただし、生成AIの出力をそのまま採用するのではなく、<br>
 実際の仕様、既存コードとの整合性、ビルド結果、動作確認を開発者側で確認したうえで反映しています。<br>
 特にゲーム進行、マップ生成、ターン管理、ユニット管理、敵及び味方AIなどの主要処理については、<br>
 生成AIの提案を参考にしつつ、最終的な実装判断と調整は開発者が行っています。<br>
